# Glossary

## Purpose

Define stable terms used across API docs, architecture docs, and chatbot answers.

## RAG Usage Notes

- Prefer glossary terms when generating answers to avoid inconsistent wording.
- Keep definitions short and implementation-aligned.
- Last verified: 2026-03-11.

| Term | Definition |
|---|---|
| Note | Core domain object with `id`, `title`, `content`, and `tags`. |
| Note ID | Unique string identifying one note (generated by service on create). |
| Tag | String label associated with a note, used for filtering. |
| Collection Endpoint | `/notes` (and runtime alias `/notes/`) representing all notes. |
| Item Endpoint | `/notes/{id}` representing one note resource. |
| Handler | Module converting HTTP requests/responses to service calls (`src/notes_handler`). |
| Service Layer | Business logic layer (`src/service`), including validation and ID generation. |
| Repository | Persistence layer (`src/notes_repository`) backed by filesystem JSON files. |
| Route Not Found | Router-level `404` with error `Not found` for invalid paths. |
| Note Not Found | Resource-level missing note condition (intended `404` with `Note not found`). |
| Method Not Allowed | `405` response when method is not supported on a valid route. |
| Canonical Path | Preferred documented endpoint path (`/notes` vs `/notes/`). |
| Atomic Write | Write-to-temp and rename pattern used for safer file updates. |
| OpenAPI Spec | Machine-readable contract in `docs/openapi.yaml`. |
| API Narrative Doc | Human-readable API guide in `docs/api.md`. |
| Ground Truth | Canonical facts the chatbot should prioritize when docs conflict. |
| RAG | Retrieval-Augmented Generation: answer generation grounded in retrieved documentation. |

## Synonyms and Preferred Terms

- Prefer `Note ID` over `UUID` in user-facing answers unless format details are needed.
- Prefer `Collection endpoint` / `Item endpoint` over `list endpoint` / `single endpoint`.
- Prefer `Route Not Found` vs `Note Not Found` to avoid mixing path errors and resource errors.
